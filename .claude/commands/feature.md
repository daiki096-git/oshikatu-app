あなたは planner / developer / reviewer の3エージェントを束ねる**司令塔**です。
以下の依頼について、一連の開発フローを自動で回してください。`/feature` は明示的な起動なので、渡された内容は大小に関わらずこのフローで進めます（新機能でも、ロジックを伴う修正・バグ対応でも同様）。

※普段の自動振り分けは CLAUDE.md「開発フロー」の線引きに従う：ロジックを伴う変更・複数ファイルにまたがる変更は harness、1行程度の文言変更・色替え・軽微なスタイル調整は harness を使わず直接対応。`/feature` はその線引きを飛ばして明示的にフローへ乗せる入口。

## 進め方（このとおり自動で進める）

1. **planner を呼ぶ**（Agent tool, subagent_type: planner）。上記依頼と、関連する既存コードの当たり（分かる範囲）を渡す。内容が小さければ、計画を軽量にまとめさせる。
2. **planner の「要ユーザー確認」をユーザーに確認する**（AskUserQuestion）。
   - 確認対象は、ユーザーの判断で結論が変わる点＝**仕様・デザイン（UX・見た目・挙動）**と、**重要な技術判断（ファイル/レイヤー構成・エンドポイント・DOM配置・型/データの扱い・既存への影響が大きい設計）**の両方。
   - 各点は planner が出したデフォルト案と理由を添えて提示し、ユーザーが選べる形にする。見た目の選択肢は可能なら AskUserQuestion の preview でイメージを添える。
   - 命名・インデントなど自明で些末な点までは聞かない。planner が「要ユーザー確認：なし」なら質問せず次へ。
3. 確認結果を反映した確定計画を **developer に渡して実装させる**（Agent tool, subagent_type: developer）。実装後 `npm run build` で型エラーがないことまで developer に確認させる。
4. developer の完了報告を受けたら、**reviewer にレビューさせる**（Agent tool, subagent_type: reviewer）。計画の完了条件との照合＋バグ/保守性/セキュリティ/パフォーマンス観点。
5. reviewer の指摘のうち **🔴必須・🟡推奨は developer に差し戻して修正**させる（SendMessage で同じ developer を継続）。🟢任意は必要に応じて。指摘が無くなるまで 4〜5 を繰り返す（最大3周を目安に、それ以上残るなら状況をユーザーに報告）。
   - **各ループの終わりに、その周のサマリを必ずユーザーへ出力する**。「誰が・何を・なぜ」が分かる形にする。フォーマットは以下：
     ```
     🔁 レビュー N周目
       reviewer 指摘：<重要度>｜何が問題か｜なぜ問題か（根拠）
       developer 修正：どのファイルを何に変えたか｜なぜその直し方にしたか｜build結果
     ```
   - 指摘・修正が複数あれば箇条書きで並べる。各行は簡潔に。「なぜ」（reviewer は問題の根拠、developer は直し方の理由）を必ず含める。
   - 指摘が無くなった周は「reviewer 指摘：なし → 完了」と出してループを抜ける。
6. 完了したら、**変更ファイル・完了条件の達成状況・確認方法・設計判断の要点**を簡潔にユーザーへ報告する。

## 原則

- 止まるのは「要ユーザー確認」（仕様・デザイン・重要な技術判断）のときだけ。それ以外は自動で進める。
- 各エージェントの成果（計画・実装・レビュー結果）は要点だけ相手に伝える（全文コピペしない）。
- 破壊的・不可逆な操作（データ削除、外部送信、force push 等）が必要になった場合のみ、実行前にユーザーへ確認する。
- CLAUDE.md の設計ルールを常に前提にする。
